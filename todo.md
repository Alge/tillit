review whether the server should also do ecosystem-specific version validation. Currently only structural payload.Validate() runs server-side; ecosystem-aware checks (Go semver, PEP 440, etc.) live only in the CLI adapters. Adding them to the server would catch malicious or buggy clients but tightly couples server to adapter packages and requires a server upgrade for every new ecosystem. Trust graph already lets peers reject signers who emit bad data, so this is defense-in-depth not security-critical.

ecosystem adapters: verify that each referenced version actually exists in the ecosystem (catches typos and hallucinated versions before they enter the trust store).

ecosystem adapters: fetch the canonical artifact hash for each version, when the ecosystem exposes one (Go's sumdb-validated zip hash, npm's integrity field, PyPI's sha256, etc). Compare against what the lockfile records and surface mismatches — protects against silent re-publication attacks where a maintainer's account is compromised and a different artifact is uploaded under the same version tag. For Go this is partly redundant with the toolchain's sumdb checks, but useful for ecosystems without an equivalent ledger.
