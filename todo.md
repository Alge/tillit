review whether the server should also do ecosystem-specific version validation. Currently only structural payload.Validate() runs server-side; ecosystem-aware checks (Go semver, PEP 440, etc.) live only in the CLI adapters. Adding them to the server would catch malicious or buggy clients but tightly couples server to adapter packages and requires a server upgrade for every new ecosystem. Trust graph already lets peers reject signers who emit bad data, so this is defense-in-depth not security-critical.

check command: optional --verify-hashes flag that fetches the canonical artifact hash for every lockfile entry via Adapter.ResolveVersion and compares it against the lockfile's recorded hash. Mismatches surfaced in the tree output as a separate marker (something distinct from [unknown]/[rejected] — maybe a trailing ⚠ "hash mismatch"). Plumbing is in place (VersionInfo.Hash); just needs the flag, the loop, and a render hook. Off by default because N HTTP calls add latency. Consider parallelising and/or caching resolved versions across runs.
