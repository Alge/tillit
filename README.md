# Tillit
A chain-of-trust signoff server for package vetting across ecosystems.

## Design

Tillit is built around a decentralized web-of-trust model:

- **Identity**: A user's username is their public key. All vetting decisions are signed with the corresponding private key, making authenticity and tamper-proofing implicit. The user's API identifier is a SHA-256 hash of their public key, encoded as base64url — this gives a fixed-size, URL-safe ID regardless of which signature algorithm is used.
- **Servers**: Anyone can self-host a Tillit server. Users register on one or more servers and upload their signed vetting decisions there.
- **Discovery**: Each peer entry in your trust list includes the server URL where that peer's signatures can be fetched. A `sync` command pulls fresh data from all peers' servers — there is no push or subscription mechanism.
- **Key management**: The CLI generates and manages keys locally. The private key never leaves the machine. Keys can be generated by the CLI or imported from an existing key. Keys are stored unencrypted on disk — security relies on OS-level disk encryption (FileVault, BitLocker, LUKS). Multiple named keys are supported, e.g. separate work and personal identities.
- **Trust graph**: Trust is transitive and depth-limited. You control how many hops you follow personally, and separately how many hops you expose to those who trust you.
- **CI/CD**: A CLI tool validates a project's lock file against a specified account's trust graph, making it easy to gate builds on organizational vetting policies.
- **Advisory, not enforcement**: Tillit is an observability and decision-support tool. It helps you make informed choices about dependencies — it does not hard-enforce anything. Distrust signals weigh more heavily than trust, but the final decision is always yours.
- **Privacy**: Trust entries have a public/private flag. Public entries are synced to your server and visible to others following your trust graph. Private entries are kept local only, so others cannot enumerate your full trust relationships by querying your server.
- **Signatures**: All vetting decisions are cryptographically signed. The signature format includes an algorithm identifier to support multiple algorithms. Default is Ed25519; SLH-DSA (SPHINCS+) is supported as a second algorithm to ensure the abstraction is sound and additional algorithms can be added easily.
- **Diff chaining**: Vetting decisions can cover a full version, a version range, or a diff between two specific versions (pinned by content hash). Trust chains can be built from these: if Lisa vetted `3.4` and Bob (trusted by you) reviewed `diff(3.4 → 3.5)`, you transitively trust `3.5`. The algorithm walks the diff chain from the last known-trusted version, stopping at the first gap or untrusted reviewer. Diff chain trust follows the same rules as people trust — no special cases.
- **Package hashing**: When signing, the CLI automatically fetches the package from the appropriate registry and computes the content hash. The adapter for each ecosystem is responsible for fetching and hashing.
- **Vetting aggregation**: When multiple trusted peers have signed the same package, the highest decision level wins. The query and check output also shows how many peers in the trust graph have vetted each package — more vettings increases confidence.
- **Conflict resolution**: Signatures are ordered by the server-assigned upload timestamp. Changing a decision requires explicitly revoking the original — uploading a conflicting decision alone is not sufficient. Rejections always take precedence over approvals regardless of ordering, for safety.
- **Decision levels**: Signatures have three levels of assurance, in ascending order of strength:
  - **Allowed** — approved for use by policy, no security review implied. Useful for organisations to define what dependencies are permitted without vouching for their safety.
  - **Vetted** — personally reviewed and confirmed safe.
  - **Rejected** — confirmed unsafe. Always takes precedence over Allowed and Vetted regardless of trust depth.
  The `check` command accepts a `--level` flag to set the minimum required level (default: Allowed).
- **Distrust**: Distrusted peers are ignored entirely and cannot contribute anywhere in the trust chain — their vettings are not inherited even transitively.
- **Cycle detection**: The trust graph resolution algorithm must handle cycles (e.g. A trusts B trusts A) with standard visited-node tracking to avoid infinite loops.
- **Project config**: A `.tillit` file at the project root overrides local defaults for that directory. The CLI walks up the directory tree to find it, similar to `.git`. Supported fields include the active key, default account for `check`, and minimum required level.

## CLI Commands

**Setup**
- `tillit init` — initialize local storage, generate keypair, optionally register on a server
- `tillit register <server_url>` — register your public key on a server

**Key management**
- `tillit key generate [--name <name>]` — generate a named keypair
- `tillit key import <path> [--name <name>]` — import an existing key
- `tillit key list` — list available keys
- `tillit key use <name>` — set the active key
- `tillit key show [--name <name>]` — print the public key for sharing with others

**Trust management**
- `tillit trust <peer@server> [--depth N] [--delegate]` — add or update a trusted peer
- `tillit distrust <peer@server>` — explicitly distrust a peer
- `tillit untrust <peer@server>` — remove a peer entirely

**Signing**
- `tillit sign version <ecosystem> <package> <version> [--level allowed|vetted|rejected]`
- `tillit sign range <ecosystem> <package> <range> [--level allowed|vetted|rejected]`
- `tillit sign diff <ecosystem> <package> <from_version> <to_version> [--level allowed|vetted|rejected]`
- `tillit revoke <signature_id>`

All signing commands accept `--key <name>` to override the active key.

**Server interaction**
- `tillit publish` — push your signatures to your server(s)
- `tillit sync` — pull fresh data from all peers' servers

**Querying**
- `tillit check <lockfile> [--account <peer@server>] [--level allowed|vetted]` — validate a lockfile against a trust graph
- `tillit query <ecosystem> <package> [--level allowed|vetted]` — show trusted versions and why
